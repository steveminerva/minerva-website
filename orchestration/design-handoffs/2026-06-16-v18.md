## ✗ Regression to fix — restricted full pages bake confidential content into static HTML
`aegis.html`, `sovereign.html`, `about.html` ship their full (now VIP/staff-only) content in the
static markup, gated **only** by CSS `visibility` during the auth check. v18 dropped the
`data-protected` injection pattern entirely (0 occurrences). A logged-out visitor can read the
content in page source.

Evidence (stripped `<main>` text length, live vs v18):

| Page | Live (secure shell) | v18 (baked) |
|---|---|---|
| `aegis.html` | ~291 chars (empty shell + `data-protected="aegis"`) | ~916 chars (full V12 specs) |
| `sovereign.html` | ~299 chars | ~1,322 chars |
| `about.html` | ~3,443 chars (public + gated Team) | ~6,956 chars |

**Decision (Steve, 2026-06-16):** Aegis / Sovereign / About-Us are **full VIP/staff-gated pages**.
The content is confidential and must not appear in page source.

## Resolution — fix in merge (no design round-trip)
1. For these 3 pages, **keep the live secure shells** (empty public shell + `<div data-protected="…">`),
   not v18's baked versions.
2. **Lift v18's new restricted copy into Supabase `protected_content`** (keys `aegis`, `sovereign`,
   `team`) so `vip.js` injects it for authorised sessions only — this preserves the new content.
3. Everything else in v18 merges as Tier 1 (public/SEO/assets) and Tier 2 (console/portal/member pages).

## Follow-up
